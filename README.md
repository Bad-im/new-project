# FireForest Monitor

Веб-модуль предназначен для загрузки спутниковых данных, запуска процедуры классификации лесного покрова и отображения карты пожароопасности.

Подробная инструкция для пользователя без опыта разработки находится в [USER_GUIDE.md](USER_GUIDE.md).

## Структура

```text
frontend/   React + TypeScript + Vite, карта на React-Leaflet
backend/    FastAPI API, заглушка сервиса классификации
```

## Запуск backend

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

Проверка:

```bash
curl http://127.0.0.1:8000/health
```

## Запуск frontend

```bash
cd frontend
npm install
npm run dev
```

Frontend по умолчанию ожидает backend по адресу `http://127.0.0.1:8000`.

Если нужен другой адрес API, создайте файл `frontend/.env`:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
```

## Возможности текущего каркаса

- Единый layout с разделами: «Обзор», «Карта», «Работа со снимками».
- Mock-авторизация на frontend: кнопка «Войти» включает режим администратора, «Выйти» возвращает гостевой режим. Состояние хранится в `localStorage`.
- Гость может открыть расширенный обзор, карту и экран требования входа для раздела «Работа со снимками».
- Администратор дополнительно может выбрать GeoTIFF, запустить демонстрационный анализ и просматривать журнал результатов.
- Карта использует Leaflet, реальные границы 23 районов, метеопрогноз Open-Meteo и фильтры отображения.
- `POST /predict` сохраняется и возвращает тестовый GeoJSON FeatureCollection.
- Backend-структура подготовлена для последующего подключения ML-модели.

## Разделы интерфейса

- `Обзор` - назначение системы, входные данные, роли, логика работы, метеомодуль, статус прототипа и ограничения.
- `Карта` - интерактивная карта с режимами «Спутниковая оценка» и «Метеопрогноз», фильтрами, popup и статистикой.
- `Работа со снимками` - защищенный раздел администратора с внутренними вкладками `Загрузка и анализ` и `Журнал результатов`.

## Роли

Гость:

- открывает сайт;
- просматривает обзор и карту;
- использует фильтры карты;
- просматривает popup-информацию по районам и метеопрогнозу;
- видит экран требования входа в разделе «Работа со снимками».

Администратор:

- имеет все возможности гостя;
- загружает GeoTIFF в тестовом интерфейсе;
- запускает тестовый анализ через `POST /predict`;
- открывает журнал результатов обработки.

## Метеомодуль и индекс Нестерова

Backend содержит дополнительный модуль кратковременной оценки пожарной опасности по метеорологическим условиям. Это не нейросетевая классификация спутниковых снимков, а расчетный прогноз на 2-3 дня по погодным данным.

Источник метеоданных:

- основной источник - Open-Meteo Forecast API;
- для расчета сухого периода дополнительно запрашиваются исторические данные Open-Meteo Archive API за последние 20 дней;
- используются hourly-параметры `temperature_2m`, `dew_point_2m`, `precipitation`;
- часовой пояс запроса - `Asia/Irkutsk`;
- для реального метеопрогноза backend должен иметь доступ к интернету и домену `api.open-meteo.com`;
- если Open-Meteo недоступен, backend не падает и возвращает mock-данные с предупреждением.

Комплексный показатель пожарной опасности Нестерова:

```text
КП = сумма t * (t - r) за последовательные сухие дни
```

где `t` - температура воздуха, `r` - температура точки росы. Если за сутки выпало `2.5 мм` осадков или больше, накопленный показатель сбрасывается.

Для кратковременного прогноза backend сначала агрегирует последние `20` исторических суток и прогноз на `3` дня. Накопление КППО начинается после последнего значимого дождя в историческом периоде (`>= 2.5 мм`). Осадки меньше `2.5 мм` не сбрасывают показатель. Если за 20 дней значимый дождь не найден, расчет продолжается по всей доступной истории, а API возвращает предупреждение:

```text
В доступной истории не найден последний значимый дождь; КППО рассчитан по доступному периоду.
```

В ответах метеомодуля дополнительно передаются поля `history_days_requested`, `history_days_used`, `last_significant_rain_date`, `last_significant_rain_mm`, `dry_period_days` и `history_used`.

Классы:

- `I`: 0-300 - отсутствует;
- `II`: 301-1000 - малая;
- `III`: 1001-4000 - средняя;
- `IV`: 4001-10000 - высокая;
- `V`: более 10000 - чрезвычайная.

Добавленные endpoint'ы:

- `GET /weather/districts` - список тестовых районов Республики Бурятия с координатами;
- `GET /weather/forecast?district=barguzinsky` - прогноз по одному району по ID или названию;
- `GET /weather/forecast/all` - прогноз по всем тестовым районам;
- `GET /weather/forecast/geojson` - прогноз в формате GeoJSON FeatureCollection для отображения на карте.
- `GET /weather/debug/open-meteo` - диагностический запрос к Open-Meteo с URL, статусом успеха и первыми значениями температуры, точки росы и осадков.

Текущие координаты районов заданы вручную как приблизительные центры. Позже их можно заменить на центроиды из shapefile или другого слоя административных границ.

## Слои карты во frontend

В разделе `Карта` используется одна общая интерактивная карта с переключаемыми режимами:

- `Спутниковая оценка` - отображает реальные границы 23 районов Республики Бурятия; нейросетевая классификация будет подключена после дообучения модели;
- `Метеопрогноз` - отображает кратковременную оценку пожарной опасности по индексу Нестерова из `/weather/forecast/all` и `/weather/forecast/geojson`;

Включение слоя метеопрогноза:

1. Откройте раздел `Карта`.
2. В блоке `Режим отображения` выберите `Метеопрогноз`.
3. В панели `Фильтры` выберите дату: сегодня, завтра или послезавтра.
4. При необходимости выберите район и нажмите `Обновить метеопрогноз`.

Спутниковая оценка и метеопрогноз решают разные задачи. Спутниковый режим сейчас показывает границы районов как основу для будущей классификации по снимкам. Метеослой показывает кратковременную расчетную опасность по температуре, точке росы, осадкам и комплексному показателю Нестерова.

## Подключение векторных границ районов

Для отображения метеопрогноза по реальным границам районов используется GeoJSON, подготовленный из shapefile.

Исходные файлы shapefile нужно положить в:

```text
backend/data/vector/districts/
```

Основной файл:

```text
backend/data/vector/districts/boundary-polygon-lvl6.shp
```

Связанные файлы `.dbf`, `.shx`, `.prj`, `.cpg` и другие должны лежать рядом с `.shp`.

Конвертация shapefile в GeoJSON:

```bash
cd backend
.venv\Scripts\activate
python scripts\convert_districts_shapefile_to_geojson.py
```

Скрипт выводит поля атрибутивной таблицы, первые названия районов, автоматически определяет поле с названием района, приводит слой к `EPSG:4326` и сохраняет результат:

```text
backend/data/geojson/districts_buryatia.geojson
```

Новые endpoint'ы:

- `GET /districts` - список районов из GeoJSON с признаком наличия геометрии и центроидом;
- `GET /districts/geojson` - полный GeoJSON с полигонами районов.

Метеопрогноз строится по всем территориям из `districts_buryatia.geojson`. В текущем наборе данных используется `23` территории Республики Бурятия. Для каждой территории backend берет центроид полигона, запрашивает Open-Meteo по координатам центроида и рассчитывает индекс Нестерова.

`GET /weather/forecast/all` возвращает прогноз по всем 23 территориям, а не по тестовому списку районов. Чтобы не делать слишком много запросов к Open-Meteo при каждом обновлении карты, прогнозы кэшируются в памяти backend на 30 минут.

`GET /weather/forecast/geojson` использует реальные полигоны из `districts_buryatia.geojson` и добавляет к ним свойства метеопрогноза: район, дату, температуру, точку росы, осадки, КППО, класс опасности, цвет и источник данных. Ответ также содержит `meta`: общее количество территорий, количество features, число районов с источником `Open-Meteo`, число mock-районов и итоговый `source` (`Open-Meteo`, `mock`, `mixed` или `unavailable`). Если район из метеомодуля не сопоставился с полигоном, он попадает в `unmatched_districts`, а mock-квадрат не создаётся.

Если GeoPandas не устанавливается на Windows, можно подготовить GeoJSON вручную через QGIS:

1. Откройте `boundary-polygon-lvl6.shp` в QGIS.
2. Проверьте кодировку слоя; при необходимости выберите UTF-8.
3. Экспортируйте слой через `Export -> Save Features As...`.
4. Выберите формат `GeoJSON` и CRS `EPSG:4326`.
5. Сохраните файл как `backend/data/geojson/districts_buryatia.geojson`.
