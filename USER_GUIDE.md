# Руководство пользователя FireForest Monitor

Это подробная инструкция для человека, который впервые открывает проект и хочет понять:

- что делает программа;
- какие файлы нужны для работы;
- как скачать, установить и запустить проект;
- как войти в систему;
- как загрузить спутниковый снимок;
- что программа считает и сохраняет;
- где смотреть результаты и как их интерпретировать;
- что делать при типичных ошибках.

## 1. Что это за программа

FireForest Monitor - это веб-приложение для оценки пожарной опасности лесного покрова.

Программа состоит из двух частей:

1. Backend - серверная часть на Python/FastAPI.
   Она принимает запросы от сайта, загружает GeoTIFF-файлы, запускает нейросетевый анализ, сохраняет результаты, рассчитывает метеорологический прогноз пожарной опасности и отдает данные для карты.

2. Frontend - пользовательский интерфейс на React/TypeScript.
   Это сайт, который открывается в браузере. Через него пользователь смотрит обзор, карту, метеопрогноз, загружает спутниковые снимки и открывает журнал результатов.

Обычно пользователь работает только с сайтом в браузере. Серверная часть должна быть запущена рядом, чтобы сайт мог получать данные.

## 2. Что умеет программа

Приложение умеет:

- показывать интерактивную карту Республики Бурятия;
- отображать границы районов;
- получать метеоданные Open-Meteo;
- рассчитывать краткосрочную пожарную опасность по индексу Нестерова;
- показывать метеопрогноз пожарной опасности на карте;
- принимать спутниковый GeoTIFF-файл;
- проверять, что снимок имеет нужный формат;
- делить снимок на фрагменты 512x512 пикселей;
- запускать нейросетевую классификацию каждого фрагмента;
- строить GeoJSON с результатами классификации;
- сохранять исходный снимок, итоговый GeoJSON, сводку и метаданные;
- открывать сохраненный анализ на карте;
- вести журнал ранее выполненных анализов;
- удалять отдельные анализы или все анализы.

## 3. Что программа не делает автоматически

Важно понимать ограничения текущей версии:

- программа не скачивает спутниковые снимки сама;
- пользователь должен заранее подготовить и загрузить GeoTIFF;
- для полноценного спутникового анализа нужен файл обученной модели `best_model.pt`;
- путь к модели сейчас задан в коде как `C:\labs\FireForest_outputs\finetune_resnet18_v2\best_model.pt`;
- если файла модели нет, раздел карты и метеопрогноза может работать, но спутниковый анализ завершится ошибкой;
- авторизация учебная: логин и пароль администратора заданы прямо в коде.

## 4. Какие данные нужно загрузить для спутникового анализа

Для анализа нужен файл GeoTIFF:

- расширение файла: `.tif` или `.tiff`;
- тип данных: геопривязанный спутниковый снимок;
- источник: Sentinel-2 или подготовленный аналогичный многоканальный снимок;
- количество каналов: ровно 7;
- порядок каналов: `B2, B3, B4, B5, B8, B11, B12`;
- размер снимка должен позволять нарезку на фрагменты 512x512;
- у файла должна быть CRS, то есть система координат;
- у файла должна быть корректная геопривязка.

Если снимок меньше 512x512 пикселей, программа не сможет получить ни одного фрагмента для анализа.

Если в снимке не 7 каналов, программа вернет ошибку вида:

```text
GeoTIFF содержит X каналов; требуется 7.
```

Если в файле нет CRS или геопривязки, программа также остановит анализ и покажет ошибку.

## 5. Что программа выгружает и сохраняет после анализа

После успешной загрузки GeoTIFF программа сохраняет результат в папку:

```text
C:\labs\FireForest_uploads\satellite_analyses
```

Для каждого анализа создается отдельная папка с уникальным идентификатором, например:

```text
20260623_120000_a1b2c3d4
```

Внутри папки сохраняются:

- `original.tif` - копия загруженного исходного GeoTIFF;
- `summary.json` - краткая сводка по анализу;
- `result.geojson` - геометрия фрагментов и класс пожароопасности для каждого фрагмента;
- `metadata.json` - служебная информация: дата анализа, имя исходного файла, путь к модели, размер фрагмента, количество обработанных фрагментов.

В интерфейсе эти данные отображаются в разделе `Работа со снимками` и на карте.

## 6. Что означает результат анализа

Программа делит снимок на фрагменты 512x512 пикселей. Каждый фрагмент классифицируется нейросетевой моделью.

Для каждого фрагмента сохраняются:

- координаты границ фрагмента;
- центр фрагмента;
- доля пригодных пикселей;
- предсказанный класс пожароопасности;
- уверенность модели;
- вероятности по каждому классу.

Классы пожароопасности:

- `1` - низкая;
- `2` - умеренная;
- `3` - средняя;
- `4` - высокая;
- `5` - чрезвычайная.

В сводке также показываются:

- `total_patches` - сколько фрагментов можно было получить из снимка;
- `processed_patches` - сколько фрагментов реально обработано;
- `skipped_patches` - сколько фрагментов пропущено;
- `dominant_class` - самый частый класс;
- `max_class` - максимальный найденный класс;
- `class_counts` - сколько фрагментов попало в каждый класс;
- `class_percentages` - распределение классов в процентах.

## 10. Как запустить backend

Backend запускается первым.

Откройте PowerShell в папке проекта и выполните:

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

После запуска сервер должен показать, что он работает на адресе:

```text
http://127.0.0.1:8000
```

Проверить сервер можно в браузере:

```text
http://127.0.0.1:8000/health
```

Также можно открыть автоматическую документацию API:

```text
http://127.0.0.1:8000/docs
```

Не закрывайте окно PowerShell с backend, пока работаете с программой.

## 11. Как запустить frontend

Frontend запускается во втором окне PowerShell.

Откройте новое окно PowerShell в папке проекта и выполните:

```powershell
cd frontend
npm install
npm run dev
```

После запуска Vite покажет адрес сайта. Обычно это:

```text
http://localhost:5173
```

Откройте этот адрес в браузере.

Если backend запущен на стандартном адресе `http://127.0.0.1:8000`, дополнительных настроек не нужно.

Если backend запущен на другом адресе, создайте файл:

```text
frontend\.env
```

И добавьте в него:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
```

После изменения `.env` перезапустите frontend.

## 13. Основные разделы сайта

В интерфейсе есть три основных раздела.

### Обзор

Раздел объясняет назначение системы, входные данные, роли пользователей и общий принцип работы.

### Карта

Раздел показывает интерактивную карту. На карте можно переключать режимы:

- спутниковая оценка;
- метеопрогноз.

В режиме метеопрогноза программа показывает расчетную пожарную опасность по районам.

В режиме спутниковой оценки можно открыть результаты загруженного GeoTIFF, если анализ уже выполнен.

### Работа со снимками

Раздел для загрузки GeoTIFF и просмотра журнала.

Внутри есть две вкладки:

- `Загрузка и анализ`;
- `Журнал результатов`.

## 14. Как выполнить спутниковый анализ

1. Запустите backend.
2. Запустите frontend.
3. Откройте сайт в браузере.
4. Войдите как администратор.
5. Откройте раздел `Работа со снимками`.
6. Откройте вкладку `Загрузка и анализ`.
7. Выберите GeoTIFF-файл `.tif` или `.tiff`.
8. Укажите дату снимка.
9. Убедитесь, что выбран набор каналов `B2, B3, B4, B5, B8, B11, B12`.
10. Нажмите `Выполнить спутниковую оценку`.
11. Дождитесь завершения обработки.
12. Нажмите `Открыть на карте`, чтобы увидеть результат.

Во время анализа backend:

- получает файл от браузера;
- временно сохраняет его;
- проверяет расширение;
- проверяет CRS, геопривязку и количество каналов;
- загружает модель ResNet18;
- режет снимок на фрагменты 512x512;
- пропускает фрагменты с недостаточным количеством пригодных пикселей;
- классифицирует пригодные фрагменты;
- формирует GeoJSON;
- сохраняет результат в папку анализов.

## 15. Как открыть старый результат

1. Откройте раздел `Работа со снимками`.
2. Перейдите во вкладку `Журнал результатов`.
3. При необходимости выберите фильтр по классу пожароопасности.
4. Нажмите `Обновить журнал`.
5. Найдите нужный анализ.
6. Нажмите кнопку открытия на карте.

Frontend загрузит сохраненный `result.geojson`, `summary.json` и `metadata.json` через backend и покажет результат на карте.

## 16. Как удалить результаты

Удаление доступно только администратору.

Можно удалить:

- один выбранный анализ;
- все сохраненные анализы.

При удалении удаляется вся папка анализа, включая:

- исходный GeoTIFF;
- summary;
- GeoJSON;
- metadata.

После удаления восстановить результат можно только повторным запуском анализа исходного снимка.

## 17. Как работает метеопрогноз

Метеопрогноз не использует нейросеть. Это отдельный расчетный модуль.

Backend получает погодные данные Open-Meteo:

- температуру воздуха;
- температуру точки росы;
- осадки.

Затем рассчитывает комплексный показатель пожарной опасности Нестерова:

```text
КППО = сумма t * (t - r) за сухие дни
```

Где:

- `t` - температура воздуха;
- `r` - температура точки росы.

Если за сутки выпало 2.5 мм осадков или больше, накопленный показатель сбрасывается.

Классы по индексу Нестерова:

- `I`: 0-300 - опасность отсутствует;
- `II`: 301-1000 - малая;
- `III`: 1001-4000 - средняя;
- `IV`: 4001-10000 - высокая;
- `V`: больше 10000 - чрезвычайная.

Если Open-Meteo недоступен, backend не должен полностью падать: он возвращает mock-данные с предупреждением.

## 18. Какие API использует программа

Основные адреса backend:

- `GET /health` - проверка, что сервер работает;
- `POST /api/auth/login` - вход администратора;
- `POST /api/satellite/analyze` - загрузка GeoTIFF и запуск анализа;
- `GET /api/satellite/analyses` - список сохраненных анализов;
- `GET /api/satellite/analyses/{analysis_id}` - подробности сохраненного анализа;
- `DELETE /api/satellite/analyses/{analysis_id}` - удалить один анализ;
- `DELETE /api/satellite/analyses` - удалить все анализы;
- `GET /districts` - список районов;
- `GET /districts/geojson` - границы районов в GeoJSON;
- `GET /weather/forecast/all` - метеопрогноз по всем районам;
- `GET /weather/forecast/geojson` - метеопрогноз в формате GeoJSON для карты;
- `GET /weather/cache/status` - состояние кэша метеопрогноза;
- `POST /weather/cache/clear` - очистка кэша метеопрогноза.

## 19. Частые ошибки и что делать

### GitHub показывает пустой репозиторий

Это значит, что репозиторий создан, но файлы еще не были успешно отправлены командой `git push`.

Возможные причины:

- не выполнена авторизация GitHub;
- GitHub отклонил слишком большой файл;
- нет remote;
- push завершился ошибкой.

### Ошибка про файл больше 100 MB

GitHub не принимает обычные файлы больше 100 MB.

В проект не нужно отправлять:

- `.log` файлы;
- временные файлы;
- кэши сборки;
- большие обучающие датасеты;
- большие модели, если они не подключены через Git LFS.

Для моделей `.pt` лучше использовать Git LFS или отдельное хранилище.

### Backend не запускается

Проверьте:

- активировано ли виртуальное окружение;
- установлены ли зависимости;
- запускаете ли команду из папки `backend`;
- свободен ли порт `8000`.

### Frontend не открывается

Проверьте:

- выполнен ли `npm install`;
- запускаете ли команду из папки `frontend`;
- свободен ли порт `5173`;
- не закрыто ли окно терминала с frontend.

### Сайт открылся, но данные не загружаются

Проверьте:

- запущен ли backend;
- открывается ли `http://127.0.0.1:8000/health`;
- совпадает ли `VITE_API_BASE_URL` с адресом backend;
- нет ли ошибок CORS в консоли браузера.

### Не получается войти

Проверьте логин и пароль:

```text
admin
123
```

Также убедитесь, что backend запущен.

### GeoTIFF не принимается

Проверьте:

- расширение `.tif` или `.tiff`;
- наличие CRS;
- наличие геопривязки;
- ровно 7 каналов;
- порядок каналов `B2, B3, B4, B5, B8, B11, B12`;
- размер снимка не меньше 512x512.

### Ошибка: файл модели не найден

Проверьте, что файл модели лежит здесь:

```text
C:\labs\FireForest_outputs\finetune_resnet18_v2\best_model.pt
```

Если путь другой, измените `MODEL_PATH` в:

```text
backend\app\services\satellite_model.py
```

### Анализ завершился, но GeoJSON пустой

Возможные причины:

- снимок меньше 512x512;
- все фрагменты были пропущены;
- слишком много пустых, нулевых или NoData-пикселей;
- снимок не соответствует ожидаемой структуре.

Полезные доработки:
